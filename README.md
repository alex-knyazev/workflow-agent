# Team Workflow Agent

Сервис автоматизированного мониторинга бэклога Jira с программируемыми правилами оповещений и доставкой уведомлений в Loop (Mattermost API).

## Стек

- Node.js 20+
- TypeScript
- ESLint
- Архитектура, вдохновлённая DDD

## Что делает

### Правило — **оповещение о не-minor баге**

- Срабатывает, когда в мониторируемом проекте появляется `Bug` с приоритетом выше `Minor`: создан, перемещён из другого проекта или повышен приоритет с Minor.
- Пропускает баги, у которых поле `Bug Environments` равно `Development`.
- Определяет получателя по стеку: `@payments-duty` для Python, `@elisov` для Android, `@knyazev.a` в остальных случаях.
- Отправляет компактное markdown-уведомление в Loop или в файл.
- После уведомления проставляет дедупликационный лейбл (`wa-non-minor-bug-alert`) на задачу Jira — при повторных опросах такие задачи пропускаются.

Формат уведомления:

```
@mention Приоритет: [KEY](jira-url/browse/KEY): Summary
стек: ...
Что нужно сделать: взять в разбор и оценить влияние на команду.
_Джейсон Стетхем говорил: ..._
```

### Правило — **ежедневный дайджест спринта**

- Запускается раз в день в 08:53 МСК по умолчанию.
- Формирует одно сообщение по задачам из активного спринта и дежурного бэкенд-спринта (`JIRA_DUTY_BACKEND_SPRINT_ID`, по умолчанию `4007`).
- Считывает спринты из кастомного поля Jira (`JIRA_SPRINT_FIELD_ID`, например `customfield_1005`).
- В сообщении: ключ задачи ссылкой, заголовок, исполнитель (`@логин`), статус.

### Правило — **очистка Sprint у отменённых задач**

- Запускается раз в день в 09:10 МСК по умолчанию.
- Делает отдельный JQL-запрос: задачи в статусе `CANCELLED`, у которых заполнено поле Sprint.
- Очищает поле Sprint для найденных задач через Jira API.
- Отправляет уведомление со списком обработанных задач.

### Правило — **оповещение о задачах в статусе Acceptance**

- Запускается каждый день в 17:50 МСК.
- Находит все задачи в статусе `Acceptance`.
- Отправляет напоминание ответственному:
  - Необходимо валидировать текущий статус и соответствие критериям приёма.
  - Если всё готово: убедиться в контакте с внешним заказчиком (если применимо), уведомить его перед переводом в Done.
  - Если нужны доработки — вернуть в разработку.
- После уведомления проставляет лейбл `wa-acceptance-status-alert` для дедупликации.

### Правило — **оповещение о Python-задачах в On Prod более 1 дня**

- Запускается каждый день в 10:00 МСК.
- Находит задачи со стеком Python, которые находятся в статусе `On Prod` более 1 дня.
- Отправляет уведомление ответственному:
  - Проверить статус задачи в production.
  - Убедиться в стабильности и отсутствии необходимости откатов.
  - Завершить задачу (перевести в Done) если всё ОК, или создать инцидент на доработку.
- После уведомления проставляет лейбл `wa-python-prod-days-alert` для дедупликации.

### Правило — **оповещение о задачах в Code Review более 1 рабочего дня**

- Запускается каждый рабочий день в 10:00 МСК.
- Находит задачи в статусе `Code Review`, для которых с момента входа в статус прошло более `8` рабочих часов.
- Рабочие часы считаются только в окне `09:00-18:00` по МСК, нерабочие дни (выходные/праздники) не учитываются.
- Отправляет уведомление исполнителю задачи (`@assignee`) с явным указанием:
  - когда задача перешла в `Code Review`;
  - сколько рабочих часов она уже находится в этом статусе.

### Правило — **оповещение о выходе в релиз (iOS/Android)**

- Срабатывает при переводе задачи со стеком iOS или Android в статус Done.
- Источник данных: только задачи Jira в статусе `Done` со стеком `Android` или `iOS`.
- В выборку попадают только тикеты, созданные после `2026-06-08 00:00`.
- Требует заполненного поля fixVersion (номер релиза).
- Определяет получателя по стеку: `@elisov` для Android, `@ios-team` для iOS.
- Отправляет компактное уведомление о выходе в релиз с номером версии.
- После уведомления проставляет лейбл `wa-mobile-release-alert` для дедупликации.

### Правило — **нормализация префикса стека в title**

- Запускается 2 раза в день: в `12:00` и `16:00` (Europe/Moscow).
- Применяется только к задачам, созданным после `2026-05-01 00:00` (Europe/Moscow).
- Обрабатывает задачи, у которых заполнено поле `Stack` и стек входит в набор: `Python`, `React`, `Android`, `iOS`.
- Приводит начало заголовка к единому формату: `[emoji Обозначение_Стека] Summary`.
  - `Python` → `[⚙️ Backend]`
  - `React` → `[⚛️ Frontend]`
  - `Android` → `[🤖 Android]`
  - `iOS` → `[🍏 iOS]`
- Для задач типа `Bug` префикс содержит два emoji: `жук + иконка стека` (например, `[🐛 ⚙️ Backend]`).
- Для `Bug` с приоритетом `Critical` или `Blocker` вместо жука ставится мигалка: `🚨` (например, `[🚨 ⚛️ Frontend]`).
- Если заголовок уже начинается со старого префикса стека (в квадратных скобках или в формате `Stack:`), старый префикс удаляется и заменяется на новый.
- Уведомление в Loop для этого правила не отправляется: правило только обновляет `summary` и проставляет лейбл `wa-stack-title-prefix-normalized`.

### Правило — **дайджест новостей Коммерсанта по платежной тематике**

- Источник данных: сайт Коммерсантъ (тема `https://www.kommersant.ru/theme/2017`).
- Запускается 2 раза в день: в 09:00 и 17:00 (Europe/Moscow).
- В 09:00 отправляет новости из окна `вчера 17:00` — `сегодня 09:00`.
- В 17:00 отправляет новости из окна `сегодня 09:00` — `сегодня 17:00`.
- Формат сообщения:
  - Заголовок: `Коммерсантъ: Платежные системы, терминалы и агенты. Новости за сегодня`
  - Далее список заголовков со ссылками без упоминаний (`@` не используется).
- Для этого правила можно задать отдельный канал Loop через `KOMMERSANT_PAYMENTS_NEWS_LOOP_CHANNEL` (например, `#team_payments`).

### Системное оповещение — **ошибка выполнения правила / фатальная ошибка процесса**

- Отправляет отдельное уведомление, если выполнение любого правила завершилось с ошибкой.
- Покрывает типичные инфраструктурные сбои (например, ошибка запроса в Jira, ошибка отправки в Loop) и необработанные исключения процесса (`uncaughtException`, `unhandledRejection`).
- Канал/DM для таких оповещений задаётся через `LOOP_FAILURE_ALERT_CHANNEL`.
- Для доставки используется Loop webhook из `LOOP_WEBHOOK_URL`.
- Чтобы избежать спама, для одного и того же rule такие уведомления отправляются не чаще одного раза в 24 часа.

## Архитектура

```
src/
  domain/        — доменные сущности и предикаты (JiraIssue, JiraSprint, …)
  application/
    ports/        — интерфейсы (JiraIssueSource, IssueLabelWriter, AlertRule, …)
    services/     — оркестрация (BacklogMonitorService, RuleConfigurationService)
    config/       — реализации правил и каталог правил
  infrastructure/
    jira/         — Jira API client
    loop/         — Loop webhook notifier
    notifier/     — FileLogNotifier (для тестового режима)
    http/         — веб-сервер управления правилами
    rules/        — хранилище override-настроек правил
  config/         — загрузка и валидация переменных окружения
public/           — статическая страница управления правилами (чистый HTML/CSS/JS, Pure CSS)
scripts/          — вспомогательные скрипты (генерация хэша пароля)
```

Каждое правило реализует интерфейс `AlertRule` с собственным расписанием (интервал или cron) и явным `источником данных`.

- Для Jira-правил источником остаются Jira-issues (active, cancelled_with_sprint, done_mobile).
- Для `kommersant-payments-news-alert` источником являются новости Коммерсанта.

Встроенный HTTP-сервер отдаёт страницу управления правилами и REST API — включить/выключить правило и изменить расписание можно без правки кода. Override-настройки сохраняются на диске и восстанавливаются при следующем запуске.

В веб-интерфейсе у каждого правила доступен блок **Тест** с двумя ручными действиями:

- **Стриггерить вручную**: немедленный запуск правила на реальных данных Jira. Если правило уже выполняется в данный момент, API вернёт ошибку.
- **Стриггерить вручную с тестовыми входящими данными**: запуск правила на заранее захардкоженном стаб-наборе Jira issue для конкретного rule (без Jira-модификаций задач).

## Переменные окружения

Скопируй `.env.example` в `.env` и заполни значения.

| Переменная                              | Обязательная    | Описание                                                                                                                     |
| --------------------------------------- | --------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| `JIRA_BASE_URL`                         | ✓               | Базовый URL инстанса Jira                                                                                                    |
| `JIRA_API_TOKEN`                        | ✓               | Bearer-токен для Jira API                                                                                                    |
| `JIRA_TEAM_PROJECT_KEYS`                | ✓               | Ключи мониторируемых проектов через запятую, например `PAY`                                                                  |
| `JIRA_STACK_FIELD_ID`                   | —               | ID кастомного поля `Стек`, например `customfield_12402`                                                                      |
| `JIRA_BUG_ENVIRONMENT_FIELD_ID`         | —               | ID кастомного поля `Bug Environments`, например `customfield_16300`                                                          |
| `JIRA_SPRINT_FIELD_ID`                  | ✓               | ID кастомного поля Sprint, например `customfield_1005`                                                                       |
| `JIRA_DUTY_BACKEND_SPRINT_ID`           | —               | ID дежурного бэкенд-спринта (по умолчанию `4007`)                                                                            |
| `NOTIFICATION_MODE`                     | —               | `loop` (по умолчанию) или `file`                                                                                             |
| `TEST_MODE`                             | —               | `true` принудительно включает режим `file`, игнорируя `NOTIFICATION_MODE`                                                    |
| `NOTIFICATION_LOG_PATH`                 | —               | Путь к файлу уведомлений в режиме `file` (по умолчанию `logs/notifications.log`)                                             |
| `LOOP_WEBHOOK_URL`                      | ✓ в loop-режиме | URL входящего вебхука Loop                                                                                                   |
| `LOOP_FAILURE_ALERT_CHANNEL`            | —               | Канал или DM Loop для алертов об ошибках выполнения правил и фатальных ошибках процесса                                      |
| `KOMMERSANT_PAYMENTS_NEWS_LOOP_CHANNEL` | —               | Канал Loop для правила новостей Коммерсанта (по умолчанию `#team_payments`)                                                  |
| `RULE_OVERRIDES_PATH`                   | —               | JSON-файл с сохранёнными override-настройками правил (по умолчанию `logs/rule-overrides.json`)                               |
| `RULE_MANAGEMENT_HOST`                  | —               | Хост веб-сервера управления правилами (по умолчанию `127.0.0.1`, или `0.0.0.0` когда платформа задаёт `PORT`)                |
| `RULE_MANAGEMENT_PORT`                  | —               | Порт веб-сервера управления правилами (по умолчанию `3210`, или значение `PORT` когда задано платформой)                     |
| `WEB_ADMIN_LOGIN`                       | ✓               | Логин единственного администратора веб-интерфейса                                                                            |
| `WEB_ADMIN_PASSWORD_HASH`               | ✓               | Хэш пароля в формате `scrypt$base64_соль$base64_хэш`                                                                         |
| `WEB_ADMIN_SESSION_TTL_MINUTES`         | —               | Время жизни сессии в минутах (по умолчанию `480`)                                                                            |
| `HOLIDAYS_DATA_PATH`                    | —               | JSON-файл с календарём праздников (по умолчанию `public/data/holidays-2026.json`). Если не найден, выходные не пропускаются. |

### Генерация хэша пароля

```bash
npm run hash:password -- "твой-пароль"
# или через stdin (безопаснее — не попадёт в историю shell)
echo -n "твой-пароль" | npm run hash:password
```

## Запуск

```bash
npm install
npm run build
npm run dev        # режим разработки с hot reload через tsx
```

Или в продакшене:

```bash
npm run start      # запускает собранный dist/main.js
```

После старта сервис выведет в консоль адрес веб-интерфейса управления правилами:

```
Team workflow agent started. Rule management UI: http://127.0.0.1:3210
```

## Docker

Для развёртывания на Ubuntu 24 удобнее всего использовать Docker-образ. Контейнер уже слушает `0.0.0.0:3210`, поэтому порт можно пробросить обычным `-p`.

1. Подготовить файл окружения на сервере. Проще всего взять `.env.example`, скопировать его в `.env` и заполнить значения.
2. Собрать образ:

```bash
docker build -t team-workflow-agent .
```

3. Запустить контейнер с пробросом порта и постоянным хранилищем для файлов состояния:

```bash
docker run -d \
  --name team-workflow-agent \
  --env-file .env \
  -p 3210:3210 \
  -v $(pwd)/logs:/app/logs \
  team-workflow-agent
```

Если хочешь использовать другой порт, задай `PORT` и синхронно измени проброс, например `-e PORT=8080 -p 8080:8080`.

После этого интерфейс будет доступен на `http://<server-ip>:3210`.

## Деплой на Railway

1. Запушить репозиторий на GitHub.
2. В Railway: `New Project` → `Deploy from GitHub repo`.
3. Настройки сервиса:
   - Build command: `npm run build`
   - Start command: `npm run start`
4. Добавить переменные окружения из таблицы выше.
5. Для боевых уведомлений установить:
   - `NOTIFICATION_MODE=loop`
   - `LOOP_WEBHOOK_URL=...`
6. Railway автоматически предоставляет `PORT` — приложение подхватывает его для `RULE_MANAGEMENT_PORT`.
7. Открыть задеплоенный URL — попадёшь на страницу управления правилами.

Чтобы проверить деплой без отправки реальных уведомлений: `TEST_MODE=true`.

## Заметки

- Все ID кастомных полей Jira берутся из env — `customfield_*` нигде не захардкожены в коде.
- Jira: только Bearer-аутентификация (`Authorization: Bearer <JIRA_API_TOKEN>`), Basic auth не используется.
- Идемпотентность через лейблы: после срабатывания правила на задачу вешается лейбл Jira — при следующих опросах задача пропускается. Правила с `skipHandledCheck: true` (дайджест, очистка спринтов) лейбл не используют.
- **Выходные и праздники**: Все правила (кроме `non-minor-bug-alert`) автоматически пропускаются по выходным (суббота, воскресенье) и в дни, указанные в календаре праздников. Праздники загружаются из JSON-файла по пути `HOLIDAYS_DATA_PATH` (по умолчанию `public/data/holidays-2026.json`). Если файл не найден, правила выполняются в обычном режиме.
- Идемпотентность через лейблы: после срабатывания правила на задачу вешается лейбл Jira — при следующих опросах задача пропускается. Правила с `skipHandledCheck: true` (дайджест, очистка спринтов) лейбл не используют. Исключение: `non-minor-bug-alert` работает независимо от выходных и праздников.
- Правило очистки спринтов делает отдельный JQL-запрос для CANCELLED-задач и не смешивает их с основной active-backlog выборкой.
- Override-настройки правил хранятся в JSON-файле на диске и восстанавливаются при перезапуске.
